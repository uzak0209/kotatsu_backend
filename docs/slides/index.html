要件

必要な性能

  • 32ms（31.25Hz）のtick rateでサーバーが状態を更新
  • 4人同時対戦
  • サーバー authoritative なアーキテクチャ
  • 低レイテンシ（目標: 20-30ms程度）

技術的課題

  • Unity クライアントからアクセス可能である必要
  • TCPでは Head-of-Line blocking が懸念
  • インフラコストを抑えたい
3

通信プロトコルの検討 ①

WebSocket (TCP)

TCP とは

Transmission Control Protocol。信頼性のある順序保証された通信。

  • パケットロスが発生したら再送
  • 順序が保証される
  • 遅延が発生しやすい（Head-of-Line blocking）

検討結果

❌ 却下

理由: TCPのHead-of-Line blockingにより、1つのパケットロスが全体の遅延を引き起こす。32msのtick rateには不十分。

Head-of-Line blocking: TCPでは、パケットが順序通りに到着する必要があるため、1つのパケットが遅延すると後続のパケットも待たされる問題。
4

通信プロトコルの検討 ②

WebTransport (HTTP/3)

WebTransport とは

HTTP/3（QUIC上）で動作する双方向通信プロトコル。ブラウザ向けに設計。

  • UDPベース
  • 複数のストリームを独立して管理
  • 低レイテンシ

検討結果

❌ 却下

理由: Unityでの公式サポートがない。サードパーティライブラリも未成熟。

5

通信プロトコルの検討 ③

UDP と QUIC の基礎

UDP とは

User Datagram Protocol。軽量で高速な通信プロトコル。

  • パケットロスしても再送しない
  • 順序保証なし
  • 低レイテンシ
  • オーバーヘッドが少ない

QUIC とは

UDPベースで、必要に応じて信頼性を提供するプロトコル。

  • 複数の独立したストリーム（Reliable）
  • Datagram（Unreliable）
  • HOL blockingを回避
  • 暗号化が標準で組み込まれている
ポイント: TCPの信頼性が必要な部分とUDPの速度が必要な部分を、1つのプロトコルで使い分けできる。
6

通信プロトコルの検討 ③

QUIC の採用決定

✓ 採用

理由: 低レイテンシを実現でき、Unity側でもネイティブUDPソケットを使用して実装可能。

通信の使い分け戦略

QUIC Stream (Reliable)
  • プレイヤーの参加通知
  • パラメータ変更
  • 重要なイベント

→ 絶対に届く必要がある通信

QUIC Datagram (Unreliable)
  • 位置座標の更新
  • リアルタイム状態同期
  • 頻繁な更新データ

→ 多少落ちても問題ない通信

インフラの検討 ①

Cloudflare Workers

メリット

  • エッジで動作、低レイテンシ
  • スケーラブル
  • 無料枠が充実
  • デプロイが簡単

デメリット

  • UDPパケットを通さない
  • WebSocket は使えるがTCPベース
  • 実行時間制限あり
❌ 却下

理由: UDPパケットが使えないため、QUICが動作しない。WebSocketはTCPベースでHOL blockingの問題が残る。

10

インフラの検討 ②

AWS (EC2)

メリット

  • 完全な制御が可能
  • UDPポート開放可能
  • スペック選択の自由度
  • スケーリング容易

デメリット

  • コストがかかる（月$10-50程度）
  • 転送量課金
  • プロトタイプには高コスト
△ 保留

理由: 技術的には問題ないが、プロトタイプ段階でコストをかけたくない。本番展開時には検討の価値あり。

11

インフラの検討 ③

オンプレミス（自宅サーバー）

課題: 動的IPアドレス

家庭用インターネット回線は通常、IPアドレスが定期的に変わる。

解決策: DDNS (Dynamic DNS)

DDNSとは、動的に変わるIPアドレスをドメイン名に自動的に紐付けるサービス。

✓ 採用

理由: コストゼロ。既存ドメインを活用。DDNSで運用可能。プロトタイプに最適。

まとめ

技術選定の流れ

  1. WebSocket (TCP) → HOL blocking で却下
  2. WebTransport → Unity サポートなしで却下
  3. QUIC → 低レイテンシ、柔軟性で採用
  4. Rust → 性能、型安全性で採用
  5. オンプレミス + DDNS → コスト最適化で採用

得られた知見

  • 通信の性質に応じて Reliable / Unreliable を使い分けることが重要
  • API と Realtime を分離し gRPC で内部通信することで責務が明確に
  • 実環境でのテストが性能保証に不可欠
  • Podman コンテナで管理することで運用が容易に
  • DDNSにより固定IPなしでも外部公開が可能

結果: 目標の32ms tickを実現し、平均20ms RTTで安定動作。Split Architecture + コンテナ化で保守性も確保。
